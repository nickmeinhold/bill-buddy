## Project Overview

Bill Buddy is a personal finance app (similar to Rocket Money) built with Flutter. It tracks subscriptions, bills, budgets, and transactions with Firebase backend.

## Current Status & Next Steps

### Recently Completed

- **Test suite & CI/CD**: Unit tests for encryption and models, GitHub Actions for PR testing and tagged deploys
- **Client-side encryption**: Wrapped key encryption (AES-256-GCM) for all financial data. Only users can decrypt their own data - not even Firebase/Google can read it.
- **Encryption auth flow**: Full flow working for both email/password and social auth (Google Sign-In):
  - New users: Set passphrase → see recovery codes → dashboard
  - Returning users: Enter passphrase → dashboard
  - Sign out: DEK cleared from memory, passphrase required on next login
- **Google Sign-In**: Working on Android and Web
- **Bank statement parsing**: Cloud Function that extracts transactions from uploaded PDF statements

### In Progress / Next Up

- **AI-powered categorization**: Integrate Gemini Flash to automatically categorize transactions from parsed bank statements
- **Multi-bank support**: Collecting PDF samples from different banks to improve parsing accuracy
- **Apple Sign-In**: Needs Apple Developer key configuration (see below)
- **Data encryption on write**: Currently encryption infrastructure is in place, but data write/read encryption needs to be integrated into the service providers

### Key Patterns

**Riverpod Providers** - Each feature has:

- `{feature}Provider` - StreamProvider watching Firestore collection
- `{feature}ServiceProvider` - Service class with CRUD operations
- `firestoreProvider` - Shared Firestore instance (in subscriptions_provider.dart)

**Models** use Equatable and include:

- `fromMap(Map, String id)` factory for Firestore
- `toMap()` for serialization
- `copyWith()` for immutability
- Computed getters (e.g., `monthlyAmount`, `yearlyAmount` on SubscriptionModel)

**Routing** uses ShellRoute for persistent bottom navigation across 5 tabs: Dashboard, Transactions, Subscriptions, Budgets, Bills.

**Encryption routing** in `app_router.dart`:

- Watches `authStateProvider`, `pendingRecoveryCodesProvider`, and `encryptionProvider.select((s) => s.isUnlocked)`
- After login, redirects to `/encryption-setup` which handles passphrase flow
- After new user setup, redirects to `/recovery-codes` before dashboard
- Uses `ref.watch` with `.select()` to avoid excessive rebuilds

### Firestore Structure

```sh

├── encryption/           # wrappedDEK, version, recoveryCodes (hashed)
├── subscriptions/{id}    # name*, amount*, frequency, nextBillingDate, category, status, notes*

├── transactions/{id}     # merchantName*, amount*, date, category, isSubscription, notes*, statementId
└── statements/{id}       # fileName, storagePath, status, transactionCount, uploadedAt

# Fields marked with * are encrypted (ENC:v1:base64...)
```

### Apple Sign In Issue (Unresolved)

Apple Sign In is implemented in code but fails with Firebase error:

```sh
[firebase_auth/invalid-credential] Invalid OAuth response from apple.com
```

**What's implemented:**

- `sign_in_with_apple` and `crypto` packages added
- `signInWithApple()` method in `AuthService` (`lib/features/auth/domain/auth_provider.dart`)
- Apple Sign In button shows on iOS/macOS in `LoginScreen`
- iOS/macOS entitlements configured for Sign in with Apple
- App ID `co.enspyr.billBuddy` has Sign in with Apple capability enabled

**The problem:**
Firebase requires the Sign in with Apple key's **Primary App ID** to directly match the app's bundle ID. Using Apple's "Group with existing primary App ID" feature does NOT work with Firebase.

**Current Apple Developer configuration:**

- App ID: `co.enspyr.billBuddy` (grouped with `co.enspyr.enspyr`)
- Services ID: `co.enspyr.billBuddy.service`
- Key: Primary App ID is `co.enspyr.enspyr` (shared across multiple apps)
- Firebase configured with Team ID, Key ID, Private Key, Services ID

**To fix:**
Create a new Sign in with Apple key with Primary App ID set directly to `co.enspyr.billBuddy` (no grouping). Apple allows up to 5 keys per account.

1. Go to <https://developer.apple.com/account/resources/authkeys/list>
2. Create new key with "Sign in with Apple" enabled
3. Set Primary App ID to `co.enspyr.billBuddy`
4. Download the .p8 file and note the Key ID
5. Update Firebase Console → Authentication → Apple provider with new Key ID and Private Key

## Firebase Setup

Requires `firebase_options.dart` generated by:

```bash
flutterfire configure
```

Firestore security rules needed:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /users/{userId}/{document=**} {
      allow read, write: if request.auth != null && request.auth.uid == userId;
    }
  }
}
```

## Client-Side Encryption

All sensitive financial data is encrypted client-side before being stored in Firestore. Even with database access, the data cannot be read without the user's password.

### Key Hierarchy

```sh
User Password ──► PBKDF2 (100k iterations) ──► KEK (Key Encryption Key)
                                                │
Random 256-bit ────────────────────────────► DEK (Data Encryption Key)
                                                │
DEK encrypted by KEK ──────────────────────► Wrapped DEK (stored in Firestore)
                                                │
Sensitive fields ◄──── AES-256-GCM ◄──────── DEK (in memory only)
```

### Key Files

- `lib/core/encryption/encryption_service.dart` - Crypto operations (PBKDF2, AES-GCM, recovery codes)
- `lib/core/encryption/encryption_provider.dart` - Riverpod state for DEK management
- `lib/features/auth/presentation/encryption_setup_screen.dart` - Handles passphrase setup/unlock for all auth types
- `lib/features/auth/presentation/encryption_passphrase_dialog.dart` - Dialog for entering/creating passphrase
- `lib/features/auth/presentation/recovery_codes_screen.dart` - Shows codes on signup (must acknowledge to continue)

### Auth Flow

1. **New email/password signup**: Password used to wrap DEK → recovery codes shown → dashboard
2. **Email/password login**: Password unwraps DEK → dashboard
3. **New social auth (Google)**: Redirected to `/encryption-setup` → prompted for passphrase → recovery codes → dashboard
4. **Returning social auth**: Redirected to `/encryption-setup` → prompted for passphrase → dashboard
5. **On logout**: DEK cleared from memory via `encryptionProvider.lock()`
6. **Recovery**: User can recover with any unused recovery code + set new passphrase

### Encrypted Fields

- Transactions: `merchantName`, `amount`, `notes`
- Subscriptions: `name`, `amount`, `notes`
- Bills: `name`, `amount`, `notes`

### parseStatement

Triggered when a PDF is uploaded to Storage. Extracts transactions and writes to Firestore.

- **Trigger**: `onObjectFinalized` for `users/{userId}/statements/*.pdf`
- **Parsing**: Currently supports Commonwealth Bank (Australia) format
- **Output**: Creates transaction documents linked via `statementId`
- **Status**: Updates statement doc with `completed`/`failed` status
